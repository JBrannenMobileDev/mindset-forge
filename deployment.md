## How mix-ups are prevented

Each repo's `firebase.json` is **pinned to exactly one site** via the
`hosting.site` field, and neither repo references the other's site:

- `~/MindsetForge/firebase.json` â†’ `"site": "mindsetforge-ai"`
- `~/mindsetforge-web/firebase.json` â†’ `"site": "mindsetforge-marketing"`
- `~/mindsetforge-admin/firebase.json` â†’ `"site": "admin-console-mindsetforge"`

Because of this, `firebase deploy --only hosting` from either repo can **only**
target its own site. The Flutter deploy script (`scripts/deploy-web.sh`) also
hard-fails if `firebase.json` is ever changed to point anywhere other than
`mindsetforge-ai`.

> âš ď¸Ź **Domain rule of thumb:** the bare apex `mindsetforge.app` is **marketing**.
> Only the `app.` subdomain is the Flutter app. When attaching a custom domain
> in the Firebase Console, double-check you selected the correct **site** first.

## Deploy the Marketing Site (`mindsetforge.app`)

```bash
cd ~/mindsetforge-web
npm run deploy
```

That runs `npm run build` (syncs published posts from Firestore, then Next.js static export) and deploys the `out/` directory.

Blog posts live at **`mindsetforge.app/blog`**. Publishing flow:

1. Drafts are generated by the `generateBlogDraft` Cloud Function (scheduled) or manually from the admin app.
2. Review/edit at **`admin.mindsetforge.app`**.
3. Click **Publish** in admin â†’ Firestore trigger fires GitHub Actions on `mindsetforge-web` â†’ site rebuilds with new static blog pages.

### Blog CI (GitHub Actions)

Workflow: `~/mindsetforge-web/.github/workflows/deploy.yml`

Triggers: `repository_dispatch` (`blog-publish`), weekly schedule, manual.

Required GitHub secrets on **`mindsetforge-web`**:

| Secret | Purpose |
|--------|---------|
| `FIREBASE_SERVICE_ACCOUNT` | JSON service account with Firestore read (build-time post sync) |
| `FIREBASE_TOKEN` | Firebase CLI deploy token |

Required Cloud Function secret on **`MindsetForge`**:

| Secret | Purpose |
|--------|---------|
| `GITHUB_DISPATCH_TOKEN` | GitHub PAT with `repo` scope to trigger the marketing deploy workflow |

## Deploy the Blog Admin (`admin.mindsetforge.app`)

```bash
cd ~/mindsetforge-admin
npm run deploy
```

One-time setup:

1. Create Firebase Auth user (email/password) for yourself.
2. Seed admin allowlist:
   ```bash
   cd ~/MindsetForge/functions
   npx ts-node scripts/seed_blog_config.ts YOUR_FIREBASE_AUTH_UID
   ```
3. Hosting site **`admin-console-mindsetforge`** serves **`admin.mindsetforge.app`** (already created).
4. Deploy Cloud Functions from `~/MindsetForge` (includes blog generator + publish trigger):
   ```bash
   cd ~/MindsetForge
   firebase deploy --only functions:generateBlogDraft,functions:createBlogDraft,functions:regenerateBlogDraft,functions:triggerBlogDeploy,functions:onBlogPostPublished,firestore:rules,firestore:indexes
   ```

## Deploy the Flutter Web App (`app.mindsetforge.app`)

```bash
cd ~/MindsetForge
./scripts/deploy-web.sh
```

That runs `flutter build web --release`, merges static assets from `public/`, then deploys.

### Build-only (no deploy)

```bash
./scripts/deploy-web.sh --build-only
```

## Custom Domain Setup (one-time, done in Firebase Console)

The `mindsetforge-marketing` hosting site needs `mindsetforge.app` connected to it.  
Do this once in the Firebase Console:

1. Go to https://console.firebase.google.com/project/mindsetforge-ai/hosting
2. Click the **mindsetforge-marketing** site â†’ **Add custom domain**
3. Enter `mindsetforge.app`
4. Follow the steps: add the verification TXT record to your DNS, then add the A records Firebase provides
5. Repeat for `www.mindsetforge.app` if desired

`app.mindsetforge.app` is connected to the `mindsetforge-ai` hosting site â€” add it the same way on **that** site's custom domain settings.

> âš ď¸Ź **This is exactly where it once broke:** `mindsetforge.app` was accidentally
> attached to the `mindsetforge-ai` (app) site, so the apex domain served the
> Flutter app instead of the marketing site. Before clicking **Add custom domain**,
> always confirm which **site** you are editing â€” the bare apex domain belongs to
> `mindsetforge-marketing`, never to `mindsetforge-ai`.
